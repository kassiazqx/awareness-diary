模块顺序 — 三层区域，sort_order 管列表
entry 展示时分三个区域：① 固定头部（entry_date + primary thread_id，直接存在 entries 表，不是模块，前端永远渲染在最顶部）；② 用户拖拽的 entry_modules，每行有 sort_order，前端按序渲染；③ 固定底部（created_at）。Todo 引用、天气、次要 Thread 引用全都是 entry_modules 里的行，可以拖动排序。
structured_data 存什么
非富文本模块用的字段，不同模块类型存不同结构：天气模块存 {"temp":22,"condition":"sunny"}；Todo引用模块存 {"todo_id":123}；Thread引用模块存 {"thread_id":45}；情绪模块存 {"score":7,"valence":"positive"}。richtext 模块的 structured_data 为 null，只用 content_json + content_plain。
Todo 完成系统 — todo_completions 是核心
todos 表只存"定义"，不存完成状态。每次完成（含补记）往 todo_completions 插一行，completed_at 可以是过去的时间，is_backdated 标记是否补记。常青型 Todo（八段锦）设 is_evergreen=true，没有截止日，永不归档，统计页直接 COUNT(*) 查 todo_completions。重复规则存 repeat_rule JSON，如 {"freq":"weekly","days":[1,3,5]}。
templates — source 字段代替两个 bool
is_system + is_user_created 两个 bool 互相冲突（用户改了系统模版算哪个？）。改为单个 source 枚举：system 是系统预设，不可编辑；user 是用户新建；cloned_from_id 记录"从哪个系统模版克隆而来"，为 null 则是从零新建。entries.template_origin_id 只是起点记录，不约束后续编辑，用于统计哪些模版最常用，以及"重置回模版"的 UX 入口。
统一 analyses 表 — 永久追加，永不覆盖，基础统计和 AI 内容分开字段
thread_analyses 并入通用 analyses 表，type 字段区分：thread（主线定期分析）/ period（周报/月报）/ tag_filter（按标签+日期范围筛查）/ custom（手动触发任意范围）。stats_json 存当时快照的基础统计（entry数/字数/情绪均值/各tag频率/打卡次数）；analysis_json + analysis_plain 存 AI 文字分析两层。每次分析都是新的一行，永不删除旧行，前端按 created_at 排列即可看到时间线演变。thread_analysis_settings 保留作为主线级触发规则，全局默认规则存在 app_settings 表。
